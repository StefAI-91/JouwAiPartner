Horizontaal of verticaal groeperen?

Nu we weten wat diepe modules zijn, komt de vraag: hoe groeperen we ze? Per technische laag (horizontaal) of per feature (verticaal)? Het antwoord is allebei — op de juiste plek.

Patroon A

Horizontaal — per laag

Alle queries bij elkaar, alle mutations bij elkaar. Zoals een fabriek met één werkplaats per type taak.

queries/ meetings · emails · themes · ...
mutations/ meetings · emails · themes · ...
actions/ meetings · emails · themes · ...
components/ meetings · emails · themes · ...
✓ Gedeelde code makkelijk importeren vanuit meerdere apps
✓ Technische conventies op één plek
✗ Feature-werk raakt 4 mappen tegelijk

De harde constraint

Waarom packages horizontaal móéten blijven

In onze monorepo worden @repo/database, @repo/ai en @repo/ui gedeeld door méérdere apps. Verticaal groeperen kán daar niet — de feature zou "vastzitten" in één app.

Stel je voor dat we verticaal zouden gaan in packages: dan zou listMeetings() in apps/cockpit/features/meetings/ leven. DevHub en de MCP-server kunnen daar niet bij — ze zijn andere apps. Horizontaal bij packages = niet-onderhandelbaar.

Stand van zaken

Hoe doen we het nu?

We zijn er deels al. Eerlijke stand per laag:

packages/database/queries/ ✓ Goed

Horizontaal. Zoals het hoort voor gedeelde code.

packages/database/mutations/ ✓ Goed

Horizontaal. Goed.

packages/ai/pipeline/ △ Shallow-drift

Horizontaal is juist — maar binnen de map zijn 20+ platte files. Zie vorige visualisatie.

cockpit/src/components/ ✓ Feature-gegroepeerd

Al verticaal: meetings/ themes/ emails/ review/ dashboard/ ...

cockpit/src/actions/ △ Hybride in uitvoering

Deels plat (organizations, people, projects), deels submap (themes/, meeting-pipeline/, meetings/). Inconsistent — maar groeit de goede kant op.

Toekomstbeeld

Hoe cockpit eruit zou kunnen zien — volledig verticaal

Let op: geen aanbeveling om nu te migreren. Dit is het denkmodel. Als je ooit merkt dat je constant zoekt naar "waar zit dat meetings-ding", is dít de vorm.

⚠️

Belangrijke nuance

Dit is geen migratie-sprint. De winst zit in consistent toepassen bij nieuwe features, niet in het herordenen van bestaande code. Blast radius van een rewrite weegt niet op tegen de ergonomische winst — zolang je de huidige structuur niet als pijnlijk ervaart.

De regel

"Verticaal als de feature geïsoleerd is in één app.
Horizontaal als de code gedeeld wordt."

Eén vraag bij elke nieuwe map: "Wordt dit door meerdere apps gebruikt?" Ja → horizontaal (per laag). Nee → verticaal (per feature).
