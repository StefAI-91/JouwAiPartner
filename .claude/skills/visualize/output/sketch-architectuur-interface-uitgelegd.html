Wat is een interface?

Eerst even wat het niet is: een scherm, een knop, een UI. Dat verwart de meeste mensen. In software-architectuur betekent "interface" iets heel anders — en met één simpele metafoor wordt het kristalhelder.

📋
In één zin
Een interface is het menu van een stuk code — wat de buitenwereld kan bestellen, niet hoe het gekookt wordt.

Hoe ze samenwerken

1. Klant leest het menu (interface) en bestelt "Burger Classic"
2. De keuken (implementatie) bakt, kookt, plateert — op z'n eigen manier
3. Klant krijgt een burger die aan het menu voldoet
De klant weet niks van de keuken. De keuken mag veranderen. Zolang de burger op het menu blijft, werkt het.

Zelfde verhaal in code

Een functie heeft ook een menu en een keuken

Neem updateThemeAction uit jullie themes-feature.

📋 INTERFACE — het "menu-item"
// Wat consumers zien:
export async function
updateThemeAction(
input: UpdateThemeInput
): Promise<ActionResult>
→ Naam: updateThemeAction
→ Input: UpdateThemeInput
→ Output: ActionResult

Dit is alles wat een consumer moet weten om de functie aan te roepen.

👨‍🍳 IMPLEMENTATIE — de "keuken"
// De body — consumer ziet dit NOOIT:
{
// 1. Zod-validatie
const parsed = updateThemeSchema.safeParse(input);
if (!parsed.success) return {...};

// 2. Admin-guard
const guard = await requireAdminInAction();
if ('error' in guard) return guard;

// 3. Supabase mutation
const result = await updateThemeMutation(...);

// 4. Revalidate paths
revalidatePath('/themes');
revalidatePath('/');

// 5. Return
return { success: true };
}

Deze 30 regels mogen ook 300 worden, of heel anders geschreven. Zolang de functie met dezelfde input dezelfde output geeft, merkt consumer er niks van.

Waarom dit ertoe doet

Jij = menu-ontwerper. AI = chef.

Dit is waarom een goede interface het verschil maakt tussen "AI werkt voor je" en "AI maakt een zooitje".

🧠

Jouw rol: menu-ontwerper

Strategisch denkwerk. Wat moet deze feature kunnen? Welke functies, welke input, welke output?

  • → "Welke acties bestaan er op themes?"
  • → "Wat gaat erin (input-schema)?"
  • → "Wat komt eruit (return type)?"
  • → "Wie mag dit doen (auth)?"
👨‍🍳

AI's rol: chef in de keuken

Tactisch werk. Vul de body van de functie in — zolang het aan het menu voldoet.

  • → Zod-validatie schrijven
  • → Database-calls maken
  • → Error-handling toevoegen
  • → revalidatePath calls
"Design the interface, delegate the implementation."
— Precies wat de talk bedoelde. Jij doet het belangrijke denkwerk. AI vult de details.

Gevolg voor testen

Proef het eten. Observeer niet de chef.

✓

Goede test

"Burger zonder kaas bestellen" → krijg ik een burger zonder kaas?

In code: gegeven deze input, is de output correct? Is er de juiste DB-call gemaakt? Heeft revalidatePath gedraaid?

✗

Slechte test (laundering)

"Heeft de chef drie keer geroerd?" — irrelevant, dat is keuken-detail.

In code: heeft deze interne helper-functie 2× geroepen? Verbreekt als je de body refactort, geeft geen extra zekerheid.

Dit is ook wat jullie CLAUDE.md al zegt: "Schrijf gedragstests, geen implementatie-tests. Assert op input→output of observable side-effects." Dat is precies het restaurant-principe: proef het eten, niet de chef.

Het komt allemaal samen

Een feature (zoals features/themes/) is een collectie van deep modules (actions, hooks, validations, components), elk met een simpele interface (een paar exports) die grote implementaties verbergen. Consumenten gebruiken alleen de interfaces. AI mag de implementaties schrijven. Jij bent de architect.

De kern

Een goede interface maakt een codebase
AI-vriendelijk.

Jij denkt (interface ontwerpen). AI doet (implementatie schrijven). Jij test (input-output). De keuken blijft verborgen.

📋 Volgende keer je een feature ontwerpt: begin met het menu, niet met de keuken.
