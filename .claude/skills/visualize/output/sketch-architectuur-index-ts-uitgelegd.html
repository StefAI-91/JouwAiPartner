Uitleg · zonder jargon

Wat is index.ts eigenlijk?

Gewoon een bestand. Maar met een speciale rol: het is het doorgeefluik van een map. Niks magisch, niks spannends. Laten we het stap voor stap afpellen.

📄
In één zin
index.ts is een bestand dat andere bestanden uit dezelfde map opnieuw naar buiten publiceert.

De metafoor

Een kamer met of zonder deur

❌ Zonder index.ts — kamer zonder deur
actions/themes/
crud.ts
review.ts
regenerate.ts

Om bij een functie te komen, moet elke consumer door de muur breken op de juiste plek. "Ik wil createTheme — die zit in crud.ts." Iedereen moet alle interne bestanden kennen.

✓ Met index.ts — één duidelijke deur
actions/themes/
crud.ts
review.ts
regenerate.ts
+ index.ts
DEUR

Consumers kloppen op één deur. index.ts regelt welke functies naar binnen/buiten gaan. Consumer hoeft niet te weten welke interne bestanden er zijn.

Het concrete verschil

Zelfde behoefte, andere import

Stel: een component wil drie functies gebruiken — createTheme, approveTheme, regenerateTheme.

❌ Zonder — 3 imports, 3 paden
// consumer moet weten waar alles zit
import {'{ createTheme }'} from '@/actions/themes/crud'
import {'{ approveTheme }'} from '@/actions/themes/review'
import {'{ regenerateTheme }'} from '@/actions/themes/regenerate'
✗3 import-regels
✗Consumer kent alle interne bestandsnamen
✗Hernoem crud.ts → alle consumers breken
✓ Met index.ts — 1 import, 1 pad
// consumer noemt alleen de map
import {'{ createTheme, approveTheme, regenerateTheme }'}
from '@/actions/themes'
✓1 import-regel
✓Consumer kent geen interne bestanden
✓Hernoem crud.ts → niemand merkt het

De inhoud

Wat staat er letterlijk in die index.ts?

Voor jullie themes/-map: drie regels. Echt. Niet meer.

// apps/cockpit/src/actions/themes/index.ts
export * from './crud'
export * from './review'
export * from './regenerate'

Wat betekent export *?

"Alles wat crud.ts naar buiten geeft, geef ik ook naar buiten." Letterlijk een doorgeefluik. crud.ts exporteert createTheme en updateTheme? Dan exporteert index.ts die nu ook — zonder dat je ze apart hoeft op te noemen.

De naam

Waarom heet het index.ts en niet bijvoorbeeld main.ts?

Het is een oude conventie, net als bij websites. Als je naar jouw-ai-partner.nl/ gaat, toont de browser automatisch index.html — zonder dat je dat in de URL hoeft te zetten.

TypeScript en Node.js doen precies hetzelfde trucje met mappen:

Website-wereld
jouw-ai-partner.nl/
→ browser opent automatisch
index.html
TypeScript-wereld
'@/actions/themes'
→ TypeScript opent automatisch
themes/index.ts

Hoe werkt het écht?

Stap voor stap: wat gebeurt er achter de schermen?

Stel: het component ThemeEditForm wil een thema bijwerken door updateTheme() aan te roepen.

1
ThemeEditForm schrijft een import
import {'{ updateTheme }'} from '@/actions/themes'

Niks bijzonders — gewoon "ik wil deze functie uit deze map".

2
TypeScript zoekt in de map

TypeScript ziet "map, geen bestand". Dus zoekt hij standaard naar themes/index.ts.

// ziet: '@/actions/themes' → map
→ open themes/index.ts
3
index.ts wijst naar de juiste file

index.ts zegt: "updateTheme? Die komt uit ./crud." (Want export * from './crud' heeft hem doorgegeven.)

→ open themes/crud.ts
→ vind updateTheme
4
Component krijgt de functie

ThemeEditForm kan nu updateTheme(...) aanroepen — zonder ooit te weten dat hij uit crud.ts komt. Het component kent alleen de voordeur.

De balans

Wat levert dit op?

🤖

Voor Claude / AI

Eén plek om te onthouden per feature. AI hoeft niet te raden welk bestand welke functie heeft. "Themes-dingen? Kijk achter de deur van themes/."

🔧

Voor refactoring

Kun je interne bestanden hernoemen of splitsen zonder dat consumers iets merken. De deur blijft hetzelfde, de kamer erachter mag veranderen.

⚖️

Wat je verliest

Consumer kan niet meer direct bij een intern bestand — móét via de deur. Dat is juist de bedoeling (geen achterdeurtjes), maar soms voelt het beperkend.

Samengevat

index.ts = doorgeefluik. Niks meer.

Een bestand van 3 regels dat zegt: "deze functies mag de buitenwereld zien, gebruik de map-naam om ze te importeren."
