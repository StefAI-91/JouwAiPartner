Losse boxen of diepe lades?

Ouster's Philosophy of Software Design zegt: de beste codebases bestaan uit weinig grote "lades" met veel verborgen complexiteit — niet uit veel kleine losse boxen. Hieronder: het principe + waar wij nu staan.

Deep: grote lades

Weinig grote lades met een simpel label. Binnenin zit dezelfde complexiteit, maar die hoef je niet te zien om hem te gebruiken.

📧 Email
📅 Meeting
📊 Weekly Summary
🧵 Themes
✓ AI kent alleen de 4 ingangen — de rest is verborgen
✓ Wijzigingen binnen een lade blijven binnen die lade
✓ Eén plek om te testen: de "deur"

Kern: dezelfde code zit in beide kasten. Het verschil is de buitenkant. Een diepe module heeft veel functionaliteit achter weinig toegangspunten. Voor AI (en voor jou) betekent dat: minder te onthouden, minder fouten, sneller begrip.

Toegepast

Hoe staat onze codebase ervoor?

Geen rewrite nodig — wel bewustzijn. We zijn op drie niveaus verschillend goed bezig.

Dit werkt al

Op package-niveau zijn we al diep. De 4 lades van onze monorepo verbergen complexiteit netjes:

@repo/database → listMeetings()
@repo/ai → processMeeting()
@repo/mcp → createMcpServer()
@repo/ui → Button, Card, ...

Een action in cockpit hoeft niet te weten hoe een meeting opgehaald wordt. Klassiek deep-module gedrag.

Dit groeit de goede kant op

In apps/cockpit/src/actions/ verschijnen submappen — de eerste stap richting diep:

actions/themes/
├─ crud.ts
├─ review.ts
└─ regenerate.ts
actions/meeting-pipeline/
├─ regenerate-meeting.ts
├─ regenerate-risks.ts
└─ reprocess-meeting.ts

Nog geen index.ts als deur — zonder die "deur" kan elke file van buiten geraakt worden. Halverwege.

Hier groeit shallow-drift

packages/ai/src/ heeft 2 mappen met veel platte files:

pipeline/ 20+ files
email-pipeline, email-pre-classifier,
email-filter-gatekeeper, summary-pipeline,
management-insights-pipeline, ...
agents/ 15+ files
gatekeeper, summarizer, classifier,
theme-detector, risk-specialist, ...

Geen facade. Consumer moet weten welke file precies. Elk nieuw agent = nóg een losse box erbij.

Het voorstel

index.ts als "deur van de lade"

Niet herstructureren als sprint. Wel deze regel toepassen zodra een domein z'n 3e of 4e bestand krijgt: één index.ts die alleen exporteert wat van buiten nodig is. Alles anders = intern.

Nu: packages/ai/src/pipeline/

Straks: gegroepeerd achter een deur

pipeline/
├─ email/

│ ├─ pre-classifier.ts
│ ├─ filter-gatekeeper.ts
│ └─ run.ts
├─ meeting/
│ ├─ index.ts ← de deur

│ └─ summarize.ts
├─ weekly/
│ └─ index.ts ← de deur
└─ themes/
└─ index.ts ← de deur
// Consumer ziet alleen de deur:
import { processEmail } from './pipeline/email'
Zoom in

Wat doet index.ts?

Het is de enige file in de map die de buitenwereld mag importeren. Hij re-exporteert wat nodig is en verbergt de rest. Zo weet AI (en jij) in één oogopslag: "Dit is wat email kan doen — de rest gaat me niets aan."

// packages/ai/src/pipeline/email/index.ts
// ✓ Wat de buitenwereld mag zien:
export { processEmail } from './run'
export type { EmailResult } from './types'
// ✗ Wat intern blijft:
// - pre-classifier.ts
// - filter-gatekeeper.ts
// - helpers.ts

De regel

"Zodra een domein z'n 3e of 4e file krijgt,
verplaats naar een submap + schrijf index.ts."

Eén kleine gewoonte. Geen Big Refactor. Met elke sprint wordt de codebase vanzelf dieper — precies wat AI nodig heeft om goed werk te leveren zonder dat jij alles in je hoofd moet houden.
